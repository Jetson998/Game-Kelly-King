# 开发步骤设计

目标：把开发拆成足够小的步骤，适合在 OpenClaw/龙虾这种上下文有限的对话里逐步推进。

## 总原则

每一轮开发只做一件事：

- 一个模块
- 一个文件或少量文件
- 一个可验证结果
- 一次小提交

避免一次性要求：

- “把整个游戏做完”
- “同时做 UI、数值、NPC、存档”
- “一次生成大量代码然后再调试”

## 阶段 0：项目准备

### Step 0.1 建立项目文件夹

输出：

- `README.md`
- `docs/` 策划与开发文档
- `src/` 开发目录
- `assets/` 资源目录

验收：目录结构清晰。

### Step 0.2 确定技术路线

推荐第一版：

- 纯 HTML + CSS + JavaScript
- 无框架
- 单文件或少量文件
- 可直接浏览器打开

理由：

- 开发快
- 上下文小
- 方便逐步调试
- 不需要复杂构建

## 阶段 1：最小可玩原型

### Step 1.1 创建静态页面骨架

文件：

- `src/index.html`
- `src/styles.css`
- `src/main.js`

功能：

- 显示标题
- 显示现金
- 显示天数
- 显示一个拍品区域
- 显示操作按钮

验收：打开页面能看到基础 UI。

### Step 1.2 加入拍品数据

文件：

- `src/data/items.js`

功能：

- 定义 20 个拍品
- 每个拍品包含名称、类型、起拍价、真实价值、描述、风险

验收：页面能随机显示一个拍品。

### Step 1.3 实现玩家出价

功能：

- 当前价
- 加价按钮
- 放弃按钮
- 玩家现金限制

验收：玩家可以加价，现金不足不能继续加。

### Step 1.4 实现简单 NPC 出价

文件：

- `src/auction.js`

功能：

- 3 个 NPC
- 每个 NPC 有最高心理价
- 轮流决定是否加价

验收：NPC 会参与竞价，拍品能成交。

### Step 1.5 实现结算与库存

功能：

- 玩家拍下后扣现金
- 拍品进入库存
- 未拍下则进入下一件

验收：库存与现金正确变化。

## 阶段 2：鉴定与出售

### Step 2.1 鉴定结果

功能：

- 拍下后显示真实价值
- 显示赚/亏预估

验收：玩家知道自己是否捡漏。

### Step 2.2 快速出售

功能：

- 可将库存物品按真实价值的一定比例出售
- 现金增加
- 库存移除

验收：能完成“拍下 → 鉴定 → 出售 → 赚钱/亏钱”的闭环。

### Step 2.3 7 天挑战

功能：

- 每天 5 件拍品
- 7 天后结算
- 达到 10000 胜利

验收：一局游戏可完整结束。

## 阶段 3：增强趣味

### Step 3.1 NPC 性格差异

功能：

- 新手小白：随机性高
- 二手贩子：估值较准
- 抬价托：会故意推价

验收：不同 NPC 行为明显不同。

### Step 3.2 心理提示

功能：

- 根据 NPC 出价行为显示文本提示
- 提示不一定 100% 准确

验收：玩家能通过提示判断是否继续竞价。

### Step 3.3 市场热度

功能：

- 每天一个热门品类
- 热门品类售价提高

验收：玩家会考虑品类和时机。

## 阶段 4：体验优化

### Step 4.1 UI 优化

- 卡牌样式
- 拍卖日志
- 成交动画/提示

### Step 4.2 平衡数值

- 调整起拍价
- 调整 NPC 预算
- 调整出售倍率
- 调整胜利目标

### Step 4.3 存档

- localStorage 保存进度
- 重新打开可继续

## 阶段 5：扩展内容

可选扩展：

- 技能树
- 多拍卖场
- 黑市
- 神秘箱子
- 每日挑战
- 排行榜
- 音效

## 每轮对话推荐格式

用户可以这样推进：

```text
继续 Step 1.1，只做页面骨架，完成后提交。
```

或者：

```text
继续 Step 1.4，只实现简单 NPC 出价，不做 UI 美化。
```

这样可以保证每次上下文都小、目标清晰、容易检查。
