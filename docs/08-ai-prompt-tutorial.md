# 用 AI 一步步做一个轻量网页小游戏：交互提示词教程

这份教程基于《湾区收买佬》的开发过程整理，目标是让别人也能用 OpenClaw / ChatGPT / Claude / Cursor 这类 AI 工具，按小步骤做出一个可玩的网页小游戏。

重点不是“一句话让 AI 生成完整游戏”，而是：

- 先定一个小而清楚的 MVP
- 把开发拆成可验证步骤
- 每轮只让 AI 做一件事
- 让 AI 写进度文档，方便下次续接
- 每步都运行检查，避免越改越乱

---

## 1. 核心方法

### 不推荐的提示词

```text
帮我做一个拍卖游戏，要好玩、完整、有 UI、有数据、有 NPC、有存档、有结算。
```

问题：范围太大，AI 容易一次生成一堆不可控代码。出了 bug 后，很难知道从哪里修。

### 推荐的提示词

```text
我要做一个轻量网页小游戏。请先不要写代码，只帮我整理：
1. 一句话概念
2. 核心循环
3. MVP 范围
4. 暂时不做什么
5. 第一版文件结构

要求：第一版只用 HTML/CSS/JavaScript，无后端，无构建步骤。
```

好处：先把范围钉住，避免 AI 一上来过度发挥。

## 2. 第 0 步：让 AI 建立项目文档

### 提示词实例

```text
请帮我建立一个轻量网页小游戏项目的文档框架。

游戏概念：玩家参加流动拍卖会，用有限现金竞拍未知价值的物品，拍下后鉴定并转卖赚钱。

请创建这些文档：
- README.md：项目说明
- docs/01-game-design.md：游戏策划案
- docs/02-development-plan.md：开发步骤
- docs/03-context-strategy.md：上下文控制方案
- docs/04-mvp-spec.md：MVP 规格
- docs/05-data-design.md：数据设计
- docs/06-task-backlog.md：任务清单
- docs/07-progress.md：开发进度

要求：
- 第一版只做浏览器单机小游戏
- 技术栈只用 HTML、CSS、JavaScript
- 不要写正式代码，只写文档和目录结构
- 开发步骤要拆得足够小，每步都有验收方式
```

### 你应该检查什么

- 文档是否真的把范围缩小了
- 是否有明确“不做什么”
- 每个开发步骤是否能在一轮对话里完成
- 是否有验收方式

## 3. 第 1 步：只做页面骨架

### 提示词实例

```text
继续 Step 1.1，只做静态页面骨架。

请先读取：
- docs/02-development-plan.md
- docs/04-mvp-spec.md

然后创建：
- src/index.html
- src/styles.css
- src/main.js

本轮只实现：
- 标题
- 现金、天数、库存上限
- 当前拍品占位卡
- 操作按钮占位
- 对手区域占位
- 拍卖日志占位

不要实现真实竞价逻辑，不要加存档，不要加复杂动画。

完成后请：
- 更新 docs/06-task-backlog.md
- 更新 docs/07-progress.md
- 告诉我如何本地打开验证
```

### 验收方式

```bash
python3 -m http.server 4173 -d src
```

打开：

```text
http://127.0.0.1:4173/
```

页面能看到基础 UI 就算完成。

## 4. 第 2 步：加入拍品数据

### 提示词实例

```text
继续 Step 1.2，只加入拍品数据和随机展示。

请先读取：
- docs/05-data-design.md
- src/index.html
- src/main.js

本轮只做：
- 创建 src/data/items.js
- 定义 20 个拍品
- 每个拍品包含：id、名称、品类、描述、起拍价、真实价值、模糊估值、风险、稀有度、标签
- 页面加载时随机显示一个拍品
- 当前价格等于起拍价

不要做玩家出价，不要做 NPC，不要做库存。

完成后请运行或说明：
- node --check src/main.js src/data/items.js
- 浏览器刷新后应能看到不同拍品
```

### 拍品数据提示词补充

如果你想让风格更明确，可以加：

```text
拍品风格请贴近深圳、香港、澳门、广州一带的现实流动货源，例如华强北电子货、港货回流、仓库尾货、旧档口库存。文案要有信息差和风险感，但不要写成玄幻或网红段子。
```

## 5. 第 3 步：实现玩家出价

### 提示词实例

```text
继续 Step 1.3，只实现玩家出价。

请先读取：
- src/index.html
- src/main.js
- src/data/items.js

本轮只做：
- 当前价格显示
- +50、+100、+500 三个加价按钮
- 玩家现金不足时禁用对应按钮
- 玩家可以放弃当前拍品
- 操作写入拍卖日志

不要做 NPC 跟价，不要做成交库存。

完成后请验证：
- 点击加价后当前价变化
- 现金不足不能继续加
- 点击放弃后按钮禁用
- node --check src/main.js src/data/items.js
```

## 6. 第 4 步：加入 NPC 竞价

### 提示词实例

```text
继续 Step 1.4，只实现简单 NPC 出价。

请先读取：
- src/main.js
- src/data/items.js
- docs/05-data-design.md

本轮创建或修改：
- src/data/npcs.js
- src/auction.js
- src/main.js

本轮只做：
- 3 个 NPC
- 每个 NPC 有不同性格和预算倾向
- 玩家出价后，NPC 轮流决定是否跟价
- NPC 出价或退出要写入日志
- 页面显示 NPC 当前状态

不要做复杂心理战，不要做市场热点，不要做存档。

完成后请验证：
- node --check src/main.js src/auction.js src/data/items.js src/data/npcs.js
- 浏览器里玩家加价后，NPC 会跟价或退出
```

### NPC 设计补充提示词

```text
NPC 不要只用“理性/冲动”这种抽象词，请给每个人一个能被玩家记住的行为特征：
- 一个容易上头
- 一个只看利润
- 一个可能故意抬价

但不要直接告诉玩家真实价值，只通过行为和短句暗示。
```

## 7. 第 5 步：成交、库存、鉴定、出售

这一块容易变大，建议拆成三轮。

### 5.1 成交与库存

```text
继续 Step 1.5，只实现成交与库存。

本轮只做：
- 所有 NPC 退出后判断最高出价者
- 如果玩家领先，扣现金并加入库存
- 如果 NPC 领先，显示被 NPC 拍走
- 如果无人出价，显示流拍
- 增加“下一件”按钮

不要做真实价值鉴定，不要做出售。

完成后更新任务清单和进度文档。
```

### 5.2 鉴定结果

```text
继续 Step 2.1，只实现鉴定结果。

本轮只做：
- 玩家拍下后显示真实价值
- 显示成交价、真实价值、预计赚亏
- 库存里也显示鉴定价和赚亏

不要做快速出售，不要做 7 天挑战。
```

### 5.3 快速出售

```text
继续 Step 2.2，只实现快速出售。

本轮只做：
- 库存物品可以快速出售
- 出售价基于真实价值的一定比例
- 出售后现金增加，库存移除
- 日志记录出售结果

不要做多渠道出售，不要做市场系统。
```

## 8. 第 6 步：做完整一局

### 提示词实例

```text
继续 Step 2.3，只实现 7 天挑战。

本轮只做：
- 每天 5 件拍品
- 共 7 天
- 结束后显示结算
- 现金达到目标算挑战成功
- 显示成交数、最终现金、库存价值、总资产

不要做成就，不要做分享，不要做动画。

完成后请更新 README、任务清单和进度文档。
```

### 建议验收

手动把每日件数临时调小，快速跑到结算页；确认：

- 天数会推进
- 拍品数量会推进
- 7 天后进入结算
- 胜负判断正确

## 9. 第 7 步：体验优化提示词

等游戏能完整玩一局后，再让 AI 做体验优化。

### UI 优化

```text
现在游戏已经能完整跑一局。请做一轮 UI 优化。

范围限制：
- 只改 src/index.html 和 src/styles.css，必要时少量改 src/main.js 的 class 名
- 不改游戏规则
- 不改拍品数据
- 不改 NPC 逻辑

目标：
- 信息层级更清楚
- 当前拍品、当前价、玩家现金更突出
- 按钮更适合手机点击
- 日志更易读

完成后请说明修改点，并运行 git diff --check。
```

### 新手教程

```text
请增加第一次进入游戏的新手引导。

范围限制：
- 只做 3 步以内
- 不要遮挡太久
- 玩家可以跳过
- 跳过后记录到 localStorage
- 设置里可以重新打开教程

教程只解释：
1. 现金目标
2. 如何看拍品判断提示
3. 如何出价或收手
```

### 移动端优化

```text
请专门优化手机窄屏体验。

目标：
- 首屏能看到现金、当前价、拍品和主操作按钮
- 次要信息可以折叠或放到下面
- 底部操作按钮固定，方便单手点击
- 不要改变桌面端布局太多

完成后请说明用哪些断点处理。
```

## 10. 第 8 步：内容扩展提示词

### 加新拍品

```text
请扩充拍品数据，从 20 个增加到 30 个。

要求：
- 保持现有字段结构不变
- 每个品类至少 5 个
- 加入高风险高回报样本
- 加入低价教学样本
- 文案保持半现代江湖拍场气质，但不要过度夸张

不要改主逻辑。
```

### 加市场热点

```text
请加入每日市场热点系统。

本轮只做：
- 每天随机一个热门品类
- 热门品类快速出售价格略高
- NPC 对热门品类略微更积极
- 页面显示今日热点和一句拍场风声

不要做复杂经济系统。
```

### 加复盘

```text
请增强结算复盘。

本轮只做：
- 最佳捡漏
- 最亏打眼
- 最佳变现
- 本局称号
- 下一局策略建议

不要做排行榜，不要接后端。
```

## 11. 通用提示词模板

### 每轮开始模板

```text
继续 [阶段 / Step]，本轮只做 [一个明确目标]。

请先读取这些文件：
- [相关文档]
- [相关代码]

范围限制：
- 做：...
- 不做：...

完成后请：
- 更新 docs/06-task-backlog.md
- 更新 docs/07-progress.md
- 运行最小验证
- 简短说明修改文件和验证结果
```

### Bug 修复模板

```text
现在有一个 bug：
[描述现象]

复现步骤：
1. ...
2. ...
3. ...

预期：
...

实际：
...

请只读取和修改相关文件，不要顺手重构其它模块。
修复后请说明原因，并运行最小验证。
```

### 控制 AI 不跑偏模板

```text
请严格控制范围：
- 不要新增框架
- 不要引入构建工具
- 不要重写整个文件
- 不要改无关文案
- 不要扩展新功能
- 只完成本轮目标
```

### 让 AI 自检的模板

```text
完成后请自检：
1. 是否只做了本轮目标？
2. 是否破坏了已有功能？
3. 是否更新了进度文档？
4. 是否有明确验证命令？
5. 如果浏览器手测，有哪些检查点？
```

## 12. 推荐开发节奏

最稳的节奏是：

1. 文档和范围
2. 静态页面
3. 拍品数据
4. 玩家出价
5. NPC 出价
6. 成交库存
7. 鉴定
8. 出售
9. 7 天结算
10. UI 优化
11. 新手引导
12. 存档
13. 移动端
14. 内容扩展
15. 发布页和反馈模板

每轮都坚持：

```text
小目标 → 小改动 → 小验证 → 写进度 → 再继续
```

这比一次性生成完整游戏慢一点，但成功率高很多，也更容易把游戏调到真正能玩。

## 13. 一整套可复制的开局提示词

如果别人想从零开始，可以直接复制下面这段：

```text
我要用 AI 一步步做一个轻量网页小游戏。请你作为开发搭档，但必须控制范围，不要一次性做完整项目。

游戏概念：玩家从小额现金起步，在流动拍卖会里竞拍未知价值的物品，拍下后鉴定并转卖。核心乐趣是低价捡漏、识别抬价、控制上头、最后赚到目标现金。

技术限制：
- 只用 HTML、CSS、JavaScript
- 无后端
- 无框架
- 无构建步骤
- 可以直接浏览器打开

请先只做项目设计和文档，不要写正式代码。

请输出并创建：
- README.md
- docs/01-game-design.md
- docs/02-development-plan.md
- docs/03-context-strategy.md
- docs/04-mvp-spec.md
- docs/05-data-design.md
- docs/06-task-backlog.md
- docs/07-progress.md

文档要求：
1. 一句话概念
2. 核心循环
3. MVP 范围
4. 明确暂时不做什么
5. 数据结构
6. 分阶段开发步骤
7. 每一步的验收方式
8. 每次对话如何续接

开发计划要拆小，每轮只改 1-4 个文件，每轮都有可验证结果。
```

## 14. 最重要的经验

- 让 AI 写代码前，先让它写边界。
- 让 AI 做功能前，先让它拆步骤。
- 每轮只做一件事。
- 文档比聊天记录可靠。
- 不要怕慢，怕的是一次生成太多，最后没人知道哪里坏了。
- 好玩的游戏不是一次生成出来的，是一轮轮试玩、修正、收窄、再加料做出来的。
