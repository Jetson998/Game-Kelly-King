# 上下文控制方案

这个项目要考虑 OpenClaw/龙虾每次对话可返回上下文有限，所以必须把信息切小、文件化、可续接。

## 1. 文件化原则

不要把所有策划、代码、任务都堆在聊天里。

所有稳定信息写入文件：

- 策划：`docs/01-game-design.md`
- 开发步骤：`docs/02-development-plan.md`
- MVP 规格：`docs/04-mvp-spec.md`
- 数据结构：`docs/05-data-design.md`
- 任务清单：`docs/06-task-backlog.md`
- 当前进度：`docs/07-progress.md`

每次继续时，只读取当前相关文件，不全文读取所有文档。

## 2. 每次开发只读必要文件

示例：

### 做 Step 1.2 拍品数据时

只需要读：

- `docs/04-mvp-spec.md`
- `docs/05-data-design.md`
- 当前相关代码文件

不需要读完整策划案。

### 做 UI 样式时

只需要读：

- `src/index.html`
- `src/styles.css`

不需要读数据文档。

## 3. 每步最大输出限制

单次开发建议限制：

- 新增/修改文件：1-4 个
- 新增代码：最好低于 300 行
- 目标：一个功能点
- 验收：一个明确结果

如果一个 Step 变大，继续拆：

```text
Step 1.4a：定义 NPC 数据结构
Step 1.4b：实现 NPC 是否出价判断
Step 1.4c：接入拍卖流程
Step 1.4d：显示 NPC 行为日志
```

## 4. 进度记录

每完成一步，更新：

- `docs/07-progress.md`

记录：

```text
日期
完成了什么
修改了哪些文件
如何验证
下一步建议
```

这样下一次不用靠聊天历史续命。

## 5. 每次任务开始模板

后续开发时，建议先写一句：

```text
继续 Step X.X，只做 XXX。请先读取相关文件，不要扩展范围。
```

## 6. 每次任务结束模板

完成后回复：

```text
已完成 Step X.X
修改文件：
- ...
验证方式：
- ...
下一步建议：
- Step X.X
```

## 7. 避免上下文爆炸的规则

- 不一次生成完整项目
- 不一次贴所有代码
- 不在回复中重复完整策划
- 大数据放 JSON/JS 文件，不放聊天
- 代码改动后用文件路径说明，不全文粘贴
- Bug 修复时只读取相关文件
- 每个阶段结束再做一次小总结

## 8. 推荐对话节奏

```text
第 1 轮：建文档和目录
第 2 轮：做页面骨架
第 3 轮：做拍品数据
第 4 轮：做玩家竞价
第 5 轮：做 NPC 竞价
第 6 轮：做成交库存
第 7 轮：做鉴定出售
第 8 轮：做 7 天挑战
第 9 轮：整体试玩和平衡
```
